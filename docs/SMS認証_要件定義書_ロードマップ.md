> 作成日: 2026-04-02
> 対象サイト: entry.reiwa-career.com
> ステータス: 計画段階

---

## 1. プロジェクト概要

### 1.1 背景

現在の「れいわキャリア」面談予約フォームでは、ユーザーが電話番号を自由に入力できる状態であり、入力された番号が本人のものであるかを確認する仕組みがない。これにより、架空の電話番号や他人の番号での登録が発生するリスクがある。

### 1.2 目的

電話番号入力後にSMS認証（ワンタイムパスワード）を導入し、**電話番号の持ち主本人であることを確認**する。これにより、登録データの信頼性を向上させ、営業担当者が架空番号に連絡するコストを削減する。

### 1.3 ゴール

- 電話番号入力（Step 6）の直後にSMS認証ステップを挟む
- 認証を通過しないと次のステップ（メールアドレス入力）に進めない
- 既存のフォームフロー・GASバックエンドへの影響を最小限に抑える

| 項目 | 内容 |
|------|------|
| フロントエンド | 静的HTML/CSS/JS（GitHub Pages） |

### 2.1 現在の電話番号入力フロー

```
Step 6: 電話番号入力
  ↓ バリデーション通過
  ↓ firstSubmit（GASへデータ送信 + Slack通知）
Step 7: メールアドレス入力へ遷移
```

### 2.2 SMS認証導入後のフロー

```
Step 6: 電話番号入力
  ↓ バリデーション通過
  ↓ 「認証コードを送信」ボタン押下
  ↓ SMS送信API呼び出し → 6桁の認証コードをSMSで送信
  ↓
Step 6.5（新規）: 認証コード入力画面
  ↓ 認証コード一致 → 認証成功
  ↓ firstSubmit（GASへデータ送信 + Slack通知）※認証済みフラグ付き
Step 7: メールアドレス入力へ遷移
```

## 3. 要件定義

### 3.1 機能要件

#### FR-1: SMS送信機能
- 入力された電話番号（日本の携帯番号: 070/080/090）に6桁の数字認証コードをSMSで送信する
- 送信元の表示名は「れいわキャリア」とする（サービスが対応している場合）

#### FR-2: 認証コード入力UI
- Step 6 と Step 7 の間に認証コード入力画面を表示する
- 6桁の数字入力フィールドを設ける
- 「認証する」ボタンで認証を実行する
- 認証成功時にのみ次のステップへ進める

#### FR-3: 認証コード検証
- ユーザーが入力したコードと送信済みコードを照合する
- 一致すれば認証成功、不一致ならエラーメッセージを表示する

#### FR-4: 再送信機能
- 「コードが届かない方はこちら」リンクで認証コードを再送信できる
- 再送信は60秒間のクールダウンを設ける（連打防止）

#### FR-5: 電話番号修正機能
- 認証コード入力画面から「電話番号を修正する」リンクで Step 6 に戻れる

#### FR-6: GAS連携
- firstSubmit のデータに `smsVerified: true` フラグを追加する
- スプレッドシートに「SMS認証済み」列を追加する

### 3.2 非機能要件

#### NFR-1: セキュリティ
- 認証コードの有効期限は **5分間**
- 認証試行回数は **最大5回**（超過時はコード無効化 + 再送信を促す）
- 認証コードはサーバーサイド（SMS配信サービス側）で生成・検証する（フロント側に正解コードを持たない）

#### NFR-2: パフォーマンス
- SMS送信から受信まで通常 **5〜10秒以内**
- 認証コード検証のレスポンスは **2秒以内**

#### NFR-3: コスト
- SMS送信単価: 約8〜15円/通（Twilio の場合）
- 月間想定コスト: 登録数 × 平均1.2通（再送信含む）
- 初期費用を最小限に抑える

#### NFR-4: ユーザビリティ
- 既存フォームのデザインテイスト（黄色グラデーション）を踏襲する
- モバイルファーストで設計する（主要ユーザーはスマホ利用を想定）
- 認証コード入力フィールドはオートフォーカスにする

## 4. SMS配信サービスの比較・選定

### 4.1 候補サービス比較

| 項目 | Twilio Verify | Firebase Auth (電話認証) |
|------|--------------|------------------------|
| 初期費用 | 無料 | 無料 |
| SMS単価 | 約$0.05/認証（約8円） | 月10,000件まで無料（※Google Cloud請求アカウント要） |
| コード生成・検証 | サービス側で完結（Verify API） | Firebase SDK側で完結 |
| GASとの連携 | REST APIで呼び出し可能 | フロントSDK中心、GAS連携は間接的 |
| セットアップ難易度 | 中（APIキー設定 + GAS側コード追加） | 中（Firebaseプロジェクト作成 + SDK導入） |
| 日本の携帯対応 | 対応 | 対応 |

### 4.2 推奨: Twilio Verify

**Twilio Verify を推奨します。** 理由は以下の通りです。

- **コード生成〜検証まで Twilio 側で完結** するため、自前でコードを保存・管理する必要がない
- GAS から REST API で直接呼び出せるため、既存のアーキテクチャ（HTML + GAS）を大きく変えずに済む
- 従量課金で、少量利用なら月額数百円〜数千円で収まる
- Verify API は認証コードの有効期限・試行回数制限が組み込み済み

### 4.3 Twilio Verify の仕組み

```
[フロント] → [GAS] → [Twilio Verify API]
    │           │            │
    │  1. 電話番号送信       │
    │  ──────────>           │
    │           │  2. SMS送信リクエスト
    │           │  ──────────>
    │           │            │  3. ユーザーにSMS送信
    │           │  <──────────
    │  <──────────           │
    │  4. 認証コード入力UI表示│
    │                        │
    │  5. コード入力・送信    │
    │  ──────────>           │
    │           │  6. コード検証リクエスト
    │           │  ──────────>
    │           │            │  7. 検証結果返却
    │           │  <──────────
    │  <──────────           │
    │  8. 認証結果に応じて処理│
```

```
sendVerificationCode(phoneNumber)
  → Twilio Verify API に SMS送信リクエスト
  → 成功/失敗を返却

checkVerificationCode(phoneNumber, code)
  → Twilio Verify API にコード検証リクエスト

```

### 5.3 スプレッドシートの変更

- P列に「SMS認証済み」列を追加（値: `TRUE` / 空白）

### 5.4 GAS のデプロイに関する注意

Twilio API の呼び出しには API キー（Account SID / Auth Token）が必要です。これらは GAS のスクリプトプロパティに保存し、コードに直接記載しないようにします。

## 10. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| SMSが届かない（キャリアブロック等） | ユーザー離脱 | 再送信機能 + 「届かない場合」の案内テキストを表示 |
| Twilio の障害 | 登録不能 | エラー時にSMS認証をスキップするフォールバック（管理者判断）を検討 |
| 認証ステップ追加による離脱率上昇 | 登録数減少 | 認証UIを極力シンプルに保つ、「30秒で完了」等の案内 |
| APIキーの漏洩 | 不正利用 | GAS スクリプトプロパティで管理、コードに直書きしない |
| 従量課金の急増 | コスト超過 | Twilio のアラート設定、月間上限の設定 |

## 11. 今後の検討事項

- **固定電話への対応**: Twilio Verify は音声通話での認証にも対応可能。将来的に固定電話ユーザー向けに音声認証を追加することも検討
- **認証済みバッジ**: スプレッドシートやSlack通知でSMS認証済みユーザーを視覚的に区別
- **重複登録の防止**: SMS認証に加えて、同一電話番号での重複登録を制限する仕組みの追加
- **reCAPTCHA の併用**: ボットによる大量SMS送信を防ぐため、SMS送信ボタン押下時にreCAPTCHA を導入

## 付録: 主要な Twilio Verify API エンドポイント

### SMS送信（Verification 作成）
```
POST https://verify.twilio.com/v2/Services/{ServiceSID}/Verifications
Body: To=+8190XXXXXXXX&Channel=sms
Auth: Basic (AccountSID:AuthToken)
```

### コード検証（VerificationCheck）
```
POST https://verify.twilio.com/v2/Services/{ServiceSID}/VerificationCheck
Body: To=+8190XXXXXXXX&Code=123456
Auth: Basic (AccountSID:AuthToken)
```

レスポンスの `status` が `"approved"` なら認証成功。
